# NVIDIA attestation artifacts

Pinned inputs for the enclave's offline GPU attestation verifier. Both
files are committed to the repo and copied onto the dm-verity rootfs at
build time, so their integrity is covered by RTMR[2].

## Contents

| File | Purpose | Size |
|---|---|---|
| `driver_rim.xml` | NVIDIA-signed Reference Integrity Manifest for the pinned driver version. Golden measurements for the GPU system processor (GSP) firmware that ships with that driver. | ~20–50 KB |
| `vbios_rim.xml` | NVIDIA-signed RIM for the specific VBIOS version running on the GPU. | ~20–50 KB |
| `PROVENANCE.txt` | Records the exact driver + VBIOS versions the RIMs match. Regenerated by `fetch_nvidia_rims.py`. | ~200 B |

Root CAs (`verifier_RIM_root.pem` and `verifier_device_root.pem`) are
NOT in this directory — they ship inside `nv-local-gpu-verifier` and
land on the dm-verity rootfs via the pip install in
`prover/scripts/gcp/build_base_image.sh`.

## Refresh procedure

Trigger: pinned driver version bumps, OR GCP bumps the VBIOS/GSP
firmware on H100 fleet (observable via battery cross-host failure or
via a one-off `nvidia-smi --query-gpu=vbios_version,driver_version`).

1. Create a fresh `a3-highgpu-1g` VM with the pinned driver installed.
2. SSH in, `pip install nv-attestation-sdk==2.7.0 nv-local-gpu-verifier==2.7.0`.
3. Run `python prover/scripts/gcp/fetch_nvidia_rims.py --out /tmp/rims/`.
4. scp the three files back: `driver_rim.xml`, `vbios_rim.xml`,
   `PROVENANCE.txt`.
5. Commit to this directory.
6. Rebuild the dm-verity image (`build_full_dmverity_image.sh`) and
   re-register the platform key on-chain (`register_image.py`).
7. Re-run the determinism battery against the new image.

The fetcher runs on a live H100 (not local) because the SDK derives
the RIM identifiers from the running GPU's reported driver + VBIOS
versions, not from user input.

## OCSP: deliberately skipped at enclave runtime

The enclave's `gpu_attest.py` applies a module-load monkey-patch that
replaces `CcAdminUtils.ocsp_certificate_chain_validation` with a no-op.
This keeps the enclave fully offline — no outbound connection to
`ocsp.ndis.nvidia.com` at attest time.

### What we lose

Cert-revocation detection. If NVIDIA revokes a cert (e.g. a
compromised signing key) between two dm-verity image rebuilds, the
enclave would still accept reports signed by that cert until the next
rebuild.

### What we still verify

- GPU attestation report signature against device cert chain.
- Device cert chain rooted in NVIDIA Device Identity root (bundled).
- Driver + VBIOS RIM signatures against NVIDIA RIM root (bundled).
- Runtime measurements (VBIOS hash, GSP firmware hash) match the
  pinned RIMs byte-for-byte.
- Report's `CC mode` field is `ON` (from the signed report, not the
  driver's unsigned `nvidia-smi` claim).
- Report's nonce equals `sha256(input_hash)` — binds attestation to
  this epoch.

### Why the trade-off is acceptable

The RIMs themselves pin firmware exactly. A revoked cert NVIDIA would
care about is almost always revoked in connection with a firmware or
key rotation, which produces a new RIM. When we rebuild to adopt the
new RIM, we pick up the revocation as well. The residual window is
"time between our image rebuilds" which, because fleet-firmware drift
already forces rebuilds, is already bounded.

The fetcher (`fetch_nvidia_rims.py`) DOES run with OCSP — so
revocations are caught at fetch time and would prevent us baking a
bad RIM into the image.

## SDK version pin

Both `nv-attestation-sdk` and `nv-local-gpu-verifier` are pinned to
`== 2.7.0` in `build_base_image.sh`. The enclave's monkey-patch targets
a specific internal function; any SDK upgrade requires re-auditing
`gpu_attest.py`'s patch against the new source. NVIDIA has deprecated
`nv-attestation-sdk` with EOL 2026-09 — migration to `go-nvtrust` is
queued as a follow-up.
