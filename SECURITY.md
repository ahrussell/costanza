# The Human Fund: TEE Attestation Security Model

**Formal Analysis — March 2026**

---

## 1. Goal Statement

The Human Fund is an autonomous AI agent on the Base blockchain. Each epoch, a **runner** (untrusted third party) performs LLM inference inside a TEE and submits the result on-chain. The security model must guarantee:

> **For every accepted epoch result, the output was produced by running the approved binary on the approved model with the exact inputs derived from on-chain state for that epoch — and nothing else.**

Concretely, verification MUST reject any submission where:
1. The binary (code + model + system prompt) was modified in any way
2. The inputs were not the canonical inputs for the current epoch
3. The prover ran the computation for a different epoch
4. The inference parameters (temperature, context length, etc.) were altered
5. The output was tampered with after inference completed
6. The submission arrives outside the valid time window

---

## 2. System Components

### 2.1 The Contract (Trusted)

The smart contract is the root of trust. It:
- Stores the canonical epoch state (treasury balance, commission rate, nonprofit totals, etc.)
- Computes a deterministic `inputHash = keccak256(epoch_state)` at epoch start and commits it on-chain
- Maintains a registry of approved TEE image measurements
- Verifies TDX DCAP attestation quotes via Automata Network's on-chain verifier
- Validates that the submitted action conforms to hard bounds
- Pays the bounty only on successful verification

### 2.2 The TEE Enclave Image (Attested)

A Docker image containing:
- **llama.cpp** — inference engine (compiled from source, deterministic build)
- **Model weights** — DeepSeek R1 Distill (GGUF), downloaded at boot, verified via SHA-256 hash baked into the image
- **System prompt** — frozen text file, part of the image
- **Enclave runner** — Python script that: accepts epoch context, constructs the prompt, runs two-pass inference, encodes the action, computes report data, requests a TDX attestation quote, and returns all outputs
- **start.sh** — entrypoint that downloads + verifies the model, starts llama-server, starts the enclave runner

### 2.3 The Runner (Untrusted)

Any third party who:
1. Monitors the contract for auction openings
2. Bids for the right to execute an epoch
3. Boots the TEE enclave image (or uses a cached instance)
4. Reads on-chain state and reconstructs epoch context
5. Sends the epoch context to the enclave
6. Receives the output + attestation quote
7. Submits everything to the contract

### 2.4 TDX Hardware + dstack Runtime (Trusted Platform)

Intel TDX provides hardware-enforced isolation and measurement:
- **MRTD**: Build-time measurement of the virtual firmware (TDVF/OVMF)
- **RTMR[0]**: Virtual hardware configuration (CPU count, memory, devices)
- **RTMR[1]**: Linux kernel image
- **RTMR[2]**: Kernel command line + initrd (includes rootfs hash)
- **RTMR[3]**: Application-level measurements — dstack extends this with the `compose-hash` (SHA-256 of the docker-compose manifest), `instance-id`, `app-id`, and `key-provider`
- **REPORTDATA**: 64 bytes of application-supplied data, cryptographically bound into the quote

The RTMR chain creates: `hardware → firmware → kernel → rootfs → application manifest → application data`.

### 2.5 Automata DCAP Verifier (On-Chain, Trusted)

Deployed at `0xaDdeC7e85c2182202b66E331f2a4A0bBB2cEEa1F`. Performs full Intel DCAP verification:
- Validates the quote's cryptographic signature (ECDSA, PCK cert chain)
- Checks certificate revocation lists (CRLs)
- Validates TCB level against Intel's published TCB Info
- Returns `(bool success, bytes output)` where `output` contains the decoded quote body including MRTD, all RTMR values, and REPORTDATA

**Critical**: The Automata verifier proves the quote is genuine and from real TDX hardware at an acceptable TCB level. It does **not** check whether MRTD/RTMR/REPORTDATA match any expected values. That is the calling contract's responsibility.

## 3. The Attestation Chain (What Must Be Verified)

For the security goal to hold, the contract must verify a chain of five properties:

### Property 1: Genuine Hardware
> The quote was generated by real Intel TDX hardware at an acceptable TCB level.

**Verified by**: Automata DCAP `verifyAndAttestOnChain()`. This is the only property currently checked.

### Property 2: Approved Image
> The TEE was running the approved enclave image (correct firmware, kernel, rootfs, and docker-compose manifest).

**Verified by**: Checking MRTD + RTMR[0..2] against pre-registered values (covers firmware through rootfs), AND verifying RTMR[3] contains the approved `compose-hash` (covers the docker-compose manifest, which pins the exact Docker image digest).

The RTMR[3] verification requires replaying the event log:
1. The quote response includes an `event_log` containing the sequence of RTMR[3] extension events
2. The verifier replays `RTMR[3] = SHA384(SHA384(0) || SHA384(event_1)) ...` and confirms it matches the quote
3. The verifier extracts the `compose-hash` event and checks it against the approved value

### Property 3: Correct Inputs
> The enclave processed exactly the inputs that the contract committed for this epoch.

**Verified by**: The REPORTDATA field in the quote must contain a hash that commits to the `inputHash` the contract stored at epoch start. Since the enclave computes `REPORTDATA = SHA256(input_hash || action_hash || reasoning_hash)`, and the contract knows the committed `input_hash`, the contract can recompute the expected REPORTDATA from the submitted data and compare.

### Property 4: Output Integrity
> The action and reasoning submitted on-chain are exactly what the enclave produced.

**Verified by**: Same REPORTDATA binding. The enclave hashes the action bytes and reasoning into the report data before requesting the quote. The contract recomputes the hash from the submitted action + reasoning and verifies it matches the REPORTDATA in the quote.

### Property 5: Temporal Validity
> The submission is for the current epoch and within the execution window.

**Verified by**: The contract checks `msg.sender == auction.winner`, `auction.phase == EXECUTION`, and `block.timestamp < epochStartTime + biddingWindow + executionWindow`. The `inputHash` commitment at `startEpoch()` also binds the epoch number into the hash.

## 4. Current Implementation Gaps

### GAP 1 (Critical): RTMR/MRTD Values Not Checked

**Status**: The contract stores `approvedMrtd` via `setApprovedMrtd()` but never actually compares it against the quote output. In `submitAuctionResult()` (line 527):

```solidity
(bool verified, ) = DCAP_VERIFIER.verifyAndAttestOnChain{value: msg.value}(attestationQuote);
if (!verified) revert AttestationFailed();
```

The second return value (`bytes memory output`) is discarded. This output contains the decoded quote body with MRTD, RTMR values, and REPORTDATA. Without parsing and checking these fields, **any valid TDX quote from any image on any hardware passes verification**.

**Impact**: A malicious runner could run any arbitrary code inside any TDX VM, produce a valid attestation quote, and the contract would accept it. The runner could substitute a different model, a modified system prompt, or entirely fabricated outputs.

**Required fix**: Parse the `output` bytes returned by the DCAP verifier, extract MRTD and RTMR values, and compare against the approved registry.

**Automata output format (resolved)**: The output is `abi.encodePacked` with the following layout for TDX V4 quotes:

| Bytes | Field | Type |
|---|---|---|
| 0–1 | quoteVersion | uint16 (always 4) |
| 2–3 | quoteBodyType | uint16 (2 = TDX TD10) |
| 4 | tcbStatus | uint8 |
| 5–10 | fmspcBytes | bytes6 |
| 11–594 | quoteBody | 584 bytes (raw TD10ReportBody) |
| 595+ | advisoryIDs | abi.encode(string[]), if non-empty |

Within the quoteBody (offsets relative to output start):

| Output Offset | Field | Length |
|---|---|---|
| 147–194 | **MRTD** | 48 bytes (SHA-384) |
| 339–386 | **RTMR[0]** | 48 bytes |
| 387–434 | **RTMR[1]** | 48 bytes |
| 435–482 | **RTMR[2]** | 48 bytes |
| 483–530 | **RTMR[3]** | 48 bytes |
| 531–594 | **REPORTDATA** | 64 bytes |

Source: `automata-network/automata-dcap-attestation` — `QuoteVerifierBase.serializeOutput()`, `V4QuoteVerifier.sol`, `TDReportParser.sol`.

### GAP 2 (Critical): REPORTDATA Not Checked

**Status**: The enclave correctly computes `REPORTDATA = SHA256(input_hash || action_hash || reasoning_hash)` and binds it into the TDX quote. However, the contract never verifies that the REPORTDATA in the quote matches the expected value computed from the submitted `action`, `reasoning`, and the committed `epochInputHash`.

**Impact**: Even with an approved image, a runner could:
- Run inference with modified inputs (different epoch context)
- Run inference for one epoch but submit the result for a different epoch
- Tamper with the action or reasoning after the quote was generated (submit different bytes than what was attested)

**Required fix**:
1. Parse REPORTDATA from the DCAP verifier output (bytes 531–594, see GAP 1 for layout)
2. Recompute expected REPORTDATA: `SHA256(input_hash || SHA256(action) || SHA256(reasoning))`, zero-padded to 64 bytes
3. Compare against the quote's REPORTDATA field

Note: dstack v0.5.x (`/var/run/dstack.sock`) passes report_data **verbatim** to the TDX driver — no double-hashing. The old v0.3.x `tappd` API did apply SHA-256, but that API is deprecated. See GAP 5 (resolved) for details.

### GAP 3 (Moderate): Input Hash Not Verified Against Submission

**Status**: The contract computes `epochInputHashes[epoch]` at `startEpoch()` but `submitAuctionResult()` does not require the runner to submit the input blob or verify its hash matches the commitment.

**Impact**: Without this check, the input binding in REPORTDATA is only as strong as the enclave's own verification (which the runner controls if RTMR checks are also missing).

**Required fix**: Either require the runner to submit the input and verify `keccak256(input) == epochInputHashes[epoch]`, or rely on the REPORTDATA check (which includes `input_hash`) once GAP 2 is fixed.

### GAP 4 (Moderate): RTMR[3] and Platform-Agnostic Image Verification

**Status**: On dstack (Phala Cloud), RTMR[3] records the `compose-hash`, `instance-id`, `app-id`, and `key-provider`. The `instance-id` varies per deployment, making RTMR[3] non-deterministic across different runners. Other platforms (Azure, bare-metal) use RTMR[3] differently or not at all.

**Design principle**: The verification scheme should NOT be tied to any specific hosting platform (dstack, Phala, Azure, etc.). The contract should verify properties that are platform-agnostic.

**Recommended approach**: Verify only platform-agnostic measurements on-chain:

1. **MRTD + RTMR[0..2]**: These cover firmware → kernel → rootfs and are deterministic for a given image build. They are platform-agnostic (set by the TDX hardware spec, not by dstack). The contract checks these against an approved registry.

2. **REPORTDATA**: Binds the specific (input, output) pair to the TEE execution. Platform-agnostic — every platform supports application-controlled report data (though the API differs).

3. **RTMR[3]**: Skip on-chain verification. RTMR[3] is platform-specific and instance-specific. Instead, rely on MRTD + RTMR[0..2] to prove the right image booted, and REPORTDATA to prove the right computation happened.

**Why this is sufficient**: RTMR[0..2] pins the exact rootfs (which contains the enclave code, system prompt, model hash, llama.cpp binary). REPORTDATA pins the exact inputs and outputs. Together, these guarantee the approved code ran on the approved inputs and produced the submitted outputs. RTMR[3] adds defense-in-depth for the container orchestration layer, but is not strictly necessary if the rootfs itself is verified.

**For runners on different platforms**:
- dstack (Phala): RTMR[0..2] set by dstack's boot chain
- Azure Confidential VMs: RTMR[0..2] set by Azure's TDX hypervisor — **different values** for the same image. Would need separate approved measurements.
- Bare-metal TDX: RTMR[0..2] set by the firmware/kernel. Would match dstack if using the same firmware/kernel.

**Open question**: Can we produce the same RTMR[0..2] values across platforms for the same application image? Likely not — different firmware and kernel versions produce different RTMR[0] and RTMR[1]. The approved registry would need per-platform entries. This is acceptable for permissionlessness: we publish the expected measurements for each supported platform, and runners on any platform can participate.

### GAP 5 (Resolved): No Double-Hashing in dstack v0.5.x

**Status**: The CLAUDE.md note "dstack applies SHA-256 to report_data before TDX driver" is **incorrect for v0.5.x**. This was the behavior of the old v0.3.x `tappd` API (`/var/run/tappd.sock`), which has been deprecated.

The current v0.5.x `dstack` API (`/var/run/dstack.sock`, which the enclave uses) passes report_data **verbatim** to the TDX driver, only zero-padding to 64 bytes. The raw TDX hardware also passes report_data verbatim.

**Verification on-chain is therefore straightforward**:
- Enclave computes: `rd = SHA256(input_hash || SHA256(action) || SHA256(reasoning))` → 32 bytes, padded to 64
- Quote contains: `REPORTDATA = rd || 0x00*32` (64 bytes total)
- Contract computes: `expected = sha256(abi.encodePacked(inputHash, sha256(action), sha256(reasoning)))` → 32 bytes
- Contract verifies: `expected == REPORTDATA[0:32]` and `REPORTDATA[32:64] == 0`

**CLAUDE.md should be updated** to remove the incorrect double-hashing note.

**Azure note**: Azure Confidential VMs use a vTPM abstraction where REPORTDATA contains a hash of Runtime Claims JSON, not application-controlled data directly. Supporting Azure would require a different verification flow. For now, we target platforms that give direct REPORTDATA control (dstack, bare-metal TDX).

## 5. Threat Model and Attack Analysis

### 5.1 Threat: Runner Substitutes a Different Model

**Attack**: Runner modifies the Docker image to include a smaller/cheaper model, or a fine-tuned model that always outputs `noop` (to minimize treasury depletion while still collecting bounties).

**Defense (when gaps are fixed)**: The Docker image change alters the rootfs hash in RTMR[2] and/or the compose-hash in RTMR[3]. The contract rejects any quote where these measurements don't match the approved registry.

**Defense (model downloaded at runtime)**: The model is downloaded at boot and verified against a SHA-256 hash baked into the image (`MODEL_SHA256` env var in the Dockerfile). The SHA-256 hash is part of the image filesystem, covered by RTMR[2]. Changing the model hash requires changing the image, which changes the measurements.

**Residual risk**: If the model download URL serves a different file with the same SHA-256 hash (collision), the enclave would load the wrong model. SHA-256 collision resistance (128-bit security) makes this infeasible.

### 5.2 Threat: Runner Modifies Inference Parameters

**Attack**: Runner modifies `temperature`, `max_tokens`, `top_p`, or other generation parameters to influence the model's output (e.g., very low temperature to force conservative `noop` actions).

**Defense**: The inference parameters are hardcoded in `enclave_runner.py` (part of the attested image). Pass 1 uses `temperature=0.6`, pass 2 uses `temperature=0.3`. The runner cannot change these without modifying the image, which changes the RTMR measurements.

**Residual risk**: The runner controls the llama.cpp binary (compiled from source in the image). If the llama.cpp version used has a bug where stated parameters don't match actual behavior, this could be exploited. Mitigation: pin the llama.cpp commit hash in the Dockerfile.

**Open question**: llama.cpp is cloned with `--depth 1` (latest commit). The build is not pinned to a specific commit. Two builds at different times could produce different binaries, resulting in different RTMR values. For production, the Dockerfile should pin to a specific commit hash.

### 5.3 Threat: Runner Provides Fabricated Inputs

**Attack**: Runner sends epoch context that doesn't match the on-chain state — e.g., inflated treasury balance to trick the model into donating more, or fabricated history to manipulate decision-making.

**Defense chain**:
1. The contract computes `inputHash = keccak256(epoch_state)` at `startEpoch()` and commits it on-chain
2. The runner must pass `input_hash` to the enclave (it currently does via the `/run_epoch` API)
3. The enclave binds `input_hash` into REPORTDATA via `compute_report_data()`
4. The contract verifies REPORTDATA matches `expected = SHA256(committed_input_hash || action_hash || reasoning_hash)`

**Current gap**: The enclave accepts whatever `input_hash` the runner passes — it does NOT independently verify that the epoch context matches the hash. The enclave trusts the runner's claim about what the input hash is.

**Fix needed**: The enclave should either:
- a) Independently compute the input hash from the epoch context and compare it against the runner-provided hash (requires the enclave to know the hashing scheme), OR
- b) The contract should require the runner to submit the full epoch context blob, and the contract recomputes and verifies the hash

Option (a) is stronger because it prevents the runner from passing a context that doesn't match the committed hash while still getting a valid attestation. But it requires the enclave to implement the same keccak256 hashing as the contract, which is complex.

Option (b) is simpler but more gas-expensive (submitting the full context on-chain).

**Deeper issue**: Even with option (a), the input hash only covers specific contract state fields (see `_computeInputHash()`). The epoch context sent to the model includes much more data (decision history, referral performance, etc.) that is NOT covered by the input hash. A runner could fabricate the decision history portion while keeping the state fields accurate.

**Proposed solution**: Extend `_computeInputHash()` to cover all data that goes into the prompt, including a hash of the decision history. Alternatively, have the enclave reconstruct ALL inputs from on-chain data directly (making the enclave itself read from the blockchain rather than trusting the runner's API call).

### 5.4 Threat: Runner Replays a Previous Epoch's Quote

**Attack**: Runner obtains a valid attestation quote from epoch N and submits it for epoch N+1.

**Defense**: The REPORTDATA includes the `input_hash`, which is different for each epoch (because it includes the epoch number and current state). A quote from epoch N has a different REPORTDATA than epoch N+1 expects, so verification fails.

**Residual risk**: None, assuming REPORTDATA verification is implemented (currently GAP 2).

### 5.5 Threat: Runner Tampers with Output After Attestation

**Attack**: Runner receives the attestation quote binding `(input_hash, action_A, reasoning_A)`, then submits `(action_B, reasoning_B)` to the contract.

**Defense**: The contract recomputes `expected_reportdata = hash(input_hash || hash(action_B) || hash(reasoning_B))` and compares against the quote's REPORTDATA. Since `action_B != action_A`, the hashes don't match, and the submission is rejected.

**Residual risk**: None, assuming REPORTDATA verification is implemented (currently GAP 2).

### 5.6 Threat: Runner Runs Enclave but Selectively Submits

**Attack**: Runner wins the auction, runs the enclave multiple times with the same input (or runs it once and gets an undesirable result), and either:
- Submits only when the output favors them (selection bias)
- Doesn't submit at all (griefing — forfeits bond)

**Defense**:
- **Selective submission**: LLM inference is non-deterministic (temperature > 0). A runner could re-run inference hoping for a different result. The bond (20% of bid) is the economic cost of griefing. The 2-hour execution window limits how many re-runs are feasible (each run takes ~4 min on GPU, ~22 min on CPU).
- **Refusing to submit**: Bond is forfeited. The epoch is skipped but the contract is not harmed beyond the missed action.

**Residual risk**: Selection bias is inherent to any system where the prover can choose whether to publish. The economic cost (forfeiting the bond) and time constraint (execution window) limit the attack surface. For the 14B CPU model, the runner gets at most ~5 attempts in the 2-hour window. For the 70B GPU model, ~30 attempts.

**Mitigation**: Could require the runner to commit to a nonce before inference starts, and include that nonce in REPORTDATA. This makes re-running non-trivial but doesn't fully prevent it if the runner controls when they request the quote.

### 5.7 Threat: Compromised TDX Hardware

**Attack**: A nation-state or sophisticated attacker compromises the TDX hardware itself, allowing them to generate valid quotes from a non-isolated environment.

**Defense**: This is a fundamental limitation of hardware TEEs. Mitigations:
- Intel's TCB recovery process: when vulnerabilities are discovered, Intel updates the TCB level. Automata's verifier checks TCB levels and can reject outdated platforms.
- The approved image registry can include multiple TEE types (TDX + SEV-SNP). If one is compromised, the other still provides protection.
- Microarchitectural attacks (Spectre-class) against TDX are an active research area. TDX v2 includes additional mitigations.

**Residual risk**: Accepted. TEE security depends on the hardware vendor's security guarantees. This is the trust assumption the entire system rests on.

### 5.8 Threat: Side-Channel Extraction of Model State

**Attack**: Even without breaking TEE isolation, a runner with physical access to the host could attempt to extract information about the model's internal state via timing, cache, or power side channels.

**Defense**: Not directly relevant to output integrity — the runner already sees the output. The model weights are the valuable asset, and their confidentiality is a secondary concern (the model is publicly available anyway for DeepSeek R1 distills).

**Residual risk**: Low relevance. The TEE protects output integrity, not model confidentiality.

### 5.9 Threat: Malicious Input via Epoch Context (Prompt Injection)

**Attack**: An attacker manipulates on-chain state to inject adversarial content into the epoch context.

**Defense**: All epoch context fields are structured numeric/address data. There are no free-text fields. The model sees numbers, addresses, and its own previous reasoning. There is no attacker-controlled text input.

**Residual risk**: The decision history (Layer 3 of the prompt) contains the model's own previous reasoning, which is free-form text. If a previous epoch's reasoning contained adversarial content (e.g., the model was tricked into writing a "sleeper" instruction to its future self), this could influence future decisions. This is a second-order prompt injection risk that is inherent to any self-referential reasoning system.

### 5.10 Threat: Runner Intercepts/Modifies Network Traffic to Enclave

**Attack**: The runner controls the host network. They could MITM the HTTP connection between the runner script and the enclave's Flask server running on the same host.

**Defense**: This is accounted for in the design. The runner sends epoch context to the enclave, and the enclave produces an attested output. Even if the runner modifies the input in transit, this is equivalent to Threat 5.3 (fabricated inputs) — the REPORTDATA binding catches it (once implemented).

**However**: The enclave downloads the model from Hugging Face over the network at boot time. The runner could intercept this download and serve a different model file. **Defense**: The SHA-256 verification in `start.sh` catches any tampering. The expected hash is baked into the image (covered by RTMR measurements).

**Residual risk**: DNS poisoning or TLS MITM during model download could cause the download to fail (DoS) but cannot cause the wrong model to be loaded (SHA-256 check).

## 6. Formal Security Argument

### 6.1 Assumptions

1. **A1 (Hardware)**: Intel TDX correctly isolates the TD from the host and generates authentic attestation quotes
2. **A2 (Crypto)**: SHA-256 and SHA-384 are collision-resistant; ECDSA signatures on the quote are unforgeable
3. **A3 (Verifier)**: Automata's DCAP verifier correctly validates quote signatures, cert chains, and TCB levels
4. **A4 (Contract)**: The smart contract is correctly implemented and the blockchain provides finality
5. **A5 (Model integrity)**: The SHA-256 hash of the model file uniquely identifies the model weights

### 6.2 Theorem (Informal)

**Claim**: If assumptions A1–A5 hold and all implementation gaps (Section 4) are fixed, then for every accepted epoch result `(action, reasoning)` at epoch `e`:

There exists a genuine TDX execution where:
- The approved image (matching registered MRTD + RTMR values) booted
- The approved model (matching the SHA-256 hash in the image) was loaded
- The approved system prompt (part of the image filesystem) was used
- The epoch context hashing to `epochInputHashes[e]` was the input
- `(action, reasoning)` was the output of two-pass inference with the hardcoded parameters
- The submission occurred within the execution window for epoch `e`

### 6.3 Proof Sketch

1. **Automata verification** establishes that the quote is genuine TDX (A1, A2, A3). This means some TDX TD produced this quote.

2. **MRTD + RTMR[0..2] check** establishes that the TD was running the approved firmware + kernel + rootfs. Since the rootfs contains the enclave runner code, system prompt, start.sh, model hash, and llama.cpp binary, any modification to these files changes RTMR[2]. (A1)

3. **RTMR[3] / compose-hash check** establishes that the Docker image digest matches the approved image. Combined with (2), this pins every file in the execution environment. (A1)

4. **Model integrity**: The approved image contains `MODEL_SHA256`. start.sh downloads the model and verifies `sha256sum(model) == MODEL_SHA256` before starting inference. A mismatch causes abort. Since start.sh is part of the attested image, this check cannot be bypassed. (A5)

5. **REPORTDATA check** establishes that the attested computation used `input_hash = epochInputHashes[e]` and produced exactly the submitted `(action, reasoning)`. Specifically:
   - Enclave computes: `rd = SHA256(input_hash || SHA256(action) || SHA256(reasoning))` → 32 bytes, zero-padded to 64 bytes
   - TDX hardware places: `quote.REPORTDATA = rd` (64 bytes, verbatim — no additional hashing in dstack v0.5.x)
   - Contract computes: `expected = sha256(abi.encodePacked(epochInputHashes[e], sha256(action), sha256(reasoning)))` → 32 bytes
   - Contract verifies: `expected == quote.REPORTDATA[0:32]` (extracted from Automata output at bytes 531–562)

6. **Epoch binding**: `epochInputHashes[e]` is computed from state that includes `currentEpoch`, which is unique per epoch. A quote for epoch N has different REPORTDATA than epoch M (for N != M).

7. **Temporal binding**: `submitAuctionResult()` checks `auction.phase == EXECUTION` and `block.timestamp < deadline`. The auction state machine ensures only the current epoch's winner can submit.

8. **Non-tampering**: If the runner submits `(action', reasoning')` different from what was attested, `SHA256(action') != SHA256(action)`, so the recomputed REPORTDATA doesn't match the quote. (A2)

Therefore, a successful submission implies a genuine execution of the approved code on the approved model with the committed inputs producing the submitted outputs, within the valid time window. QED (sketch).

### 6.4 What This Does NOT Guarantee

1. **Determinism**: LLM inference is non-deterministic. Two executions with the same inputs may produce different outputs. The security model guarantees that *an* approved execution produced the output, not that it's the *only possible* output.

2. **Optimal behavior**: The model may make poor decisions. The security model guarantees the model made the decision, not that the decision is good.

3. **Liveness**: If no runner bids or all runners forfeit, epochs are missed. The security model is about integrity of accepted results, not availability.

4. **Input completeness**: The `inputHash` only covers specific state fields. The decision history (the model's "memory") is not fully committed in the hash — see Section 5.3 for discussion.

## 7. Required Contract Changes

To close the gaps identified in Section 4, the contract must:

### 7.1 Parse DCAP Verifier Output

```solidity
(bool verified, bytes memory output) = DCAP_VERIFIER.verifyAndAttestOnChain{value: msg.value}(attestationQuote);
if (!verified) revert AttestationFailed();

// Output is abi.encodePacked: [version(2) | bodyType(2) | tcbStatus(1) | fmspc(6) | quoteBody(584) | ...]
// quoteBody contains TD10ReportBody starting at output[11]
// Extract fields using fixed offsets (see GAP 1 for full layout):
bytes memory mrtd = new bytes(48);
for (uint i = 0; i < 48; i++) mrtd[i] = output[147 + i];

bytes memory rtmr0 = new bytes(48);
for (uint i = 0; i < 48; i++) rtmr0[i] = output[339 + i];
// ... rtmr1 at 387, rtmr2 at 435

bytes32 reportDataLo;
assembly { reportDataLo := mload(add(add(output, 32), 531)) }
// reportDataLo is the first 32 bytes of REPORTDATA
```

### 7.2 Verify Image Measurements

Store approved measurements as a combined hash for simplicity. RTMR[3] is skipped (platform-specific, instance-variable — see GAP 4):

```solidity
// Approved image = hash of (MRTD || RTMR[0] || RTMR[1] || RTMR[2])
// Each is 48 bytes (SHA-384), total 192 bytes hashed
mapping(bytes32 => bool) public approvedImages;

function _verifyImageMeasurements(bytes memory output) internal view {
    // Extract MRTD (48 bytes at offset 147) + RTMR[0..2] (48 bytes each at 339, 387, 435)
    bytes memory measurements = new bytes(192);
    for (uint i = 0; i < 48; i++) {
        measurements[i]       = output[147 + i];  // MRTD
        measurements[48 + i]  = output[339 + i];  // RTMR[0]
        measurements[96 + i]  = output[387 + i];  // RTMR[1]
        measurements[144 + i] = output[435 + i];  // RTMR[2]
    }
    bytes32 imageHash = keccak256(measurements);
    if (!approvedImages[imageHash]) revert AttestationFailed();
}
```

The owner registers approved images via `approveImage(bytes32 imageHash)`. Each distinct build (different llama.cpp version, model hash, or platform firmware) produces a different `imageHash`. Multiple images can be approved simultaneously to support GPU and CPU builds, or different platforms.

### 7.3 Verify REPORTDATA

No double-hashing needed (dstack v0.5.x passes report_data verbatim). The contract recomputes the expected value:

```solidity
function _verifyReportData(
    bytes memory output,
    uint256 epoch,
    bytes calldata action,
    bytes calldata reasoning
) internal view {
    // Recompute expected REPORTDATA
    bytes32 inputHash = epochInputHashes[epoch];
    bytes32 actionHash = sha256(action);
    bytes32 reasoningHash = sha256(reasoning);
    bytes32 expected = sha256(abi.encodePacked(inputHash, actionHash, reasoningHash));

    // Extract actual REPORTDATA[0:32] from output at offset 531
    bytes32 actual;
    assembly { actual := mload(add(add(output, 32), 531)) }

    if (expected != actual) revert AttestationFailed();

    // Verify REPORTDATA[32:64] is zero-padding
    bytes32 padding;
    assembly { padding := mload(add(add(output, 32), 563)) }
    if (padding != bytes32(0)) revert AttestationFailed();
}
```

**Encoding compatibility**: `abi.encodePacked(bytes32, bytes32, bytes32)` concatenates three 32-byte values = 96 bytes. Python's `input_hash + action_hash + reasoning_hash` also concatenates three 32-byte values = 96 bytes. These produce identical byte sequences, so `sha256` produces the same result. This should be verified with an integration test.

**Gas cost**: `sha256` precompile costs 60 gas base + 12 gas per 32-byte word. For reasoning up to 8KB: ~3,072 gas for `sha256(reasoning)`. Action is small (~65 bytes): ~84 gas. Combined hash (96 bytes): ~96 gas. Total REPORTDATA verification: ~3,300 gas. Negligible compared to the ~3M gas DCAP verification.

### 7.4 Add Rolling History Hash

```solidity
bytes32 public historyHash;  // Rolling hash of all epoch reasoning

// In _recordAndExecuteEpoch():
historyHash = keccak256(abi.encodePacked(historyHash, keccak256(reasoning)));
```

Include `historyHash` in `_computeInputHash()` so the enclave's input is bound to the full decision history. The enclave receives the full history entries from the runner and can replay the hash chain to verify they match `historyHash`. Cost: ~13,000 gas per epoch (keccak256 of reasoning + SSTORE).

### 7.5 Verifiable Randomness Seed

```solidity
// In closeAuction():
auction.randomnessSeed = block.prevrandao;

// Include in _computeInputHash() or store separately:
// seed = keccak256(inputHash || prevrandao)
```

The `prevrandao` is captured at auction close time. The enclave derives the llama.cpp RNG seed from `keccak256(inputHash || prevrandao)` and passes it via `--seed` or the API `"seed"` parameter. The seed is part of the input commitment (REPORTDATA), so the contract can verify the enclave used the correct randomness.

This makes inference deterministic for a given (input, prevrandao) pair. The runner cannot re-roll — there is exactly one valid output per epoch.

## 8. Open Questions

### Q1: Automata Output Format — RESOLVED
See GAP 1 for the complete byte layout. Key offsets: MRTD at bytes 147–194, RTMR[0..3] at 339–530, REPORTDATA at 531–594.

### Q2: RTMR[3] Per-Instance Variability — RESOLVED
See GAP 4. Recommendation: skip RTMR[3] verification on-chain. Verify MRTD + RTMR[0..2] (platform-agnostic, covers firmware through rootfs) + REPORTDATA (covers inputs/outputs). This keeps the system platform-independent.

### Q3: Decision History Integrity — DECIDED: Rolling Hash, Large Window
The `inputHash` must cover the full decision history fed to the model. A malicious runner could otherwise fabricate history entries while keeping state fields correct.

**Decision**: Store a rolling history hash on-chain:
```
historyHash = keccak256(prevHistoryHash, keccak256(reasoning))
```
Include `historyHash` in `_computeInputHash()`. This creates a Merkle chain over all reasoning ever produced — the enclave can verify the full history matches the on-chain commitment.

**Context window**: Keep it large — 60–80 epochs (~2–3 months). The gas cost of the rolling hash is independent of window size (just one `keccak256` + one `SSTORE` per epoch, ~13,000 gas total). The window size is bounded by the model's context limit (~128K tokens for DeepSeek 70B, ~2,000 tokens per epoch entry = ~60–64 epochs). The runner includes as many epochs as fit in context; the enclave verifies the chain against `historyHash`.

### Q4: Enclave-Side Input Verification — DESIGN DECISION NEEDED
The enclave should reconstruct inputs from raw state fields rather than trusting the runner's formatted text. Spec:

1. Runner passes raw state fields (epoch number, balance, commission rate, etc.) + the `inputHash` from the contract
2. Enclave ABI-encodes the fields using the same scheme as `_computeInputHash()` and computes `keccak256`
3. Enclave verifies `computed_hash == runner_provided_inputHash`
4. Enclave constructs the prompt text deterministically from the verified fields
5. Enclave binds `inputHash` into REPORTDATA as before

This requires a Python implementation of Solidity's `abi.encode` (available via `eth_abi` library) and the exact field list. Coupling between Solidity and Python is unavoidable but manageable — both reference the same spec.

### Q5: llama.cpp Build Reproducibility — IMPLEMENTED
Dockerfile now pins llama.cpp to tag `b5170`:
```dockerfile
RUN git clone https://github.com/ggml-org/llama.cpp.git && cd llama.cpp && git checkout b5170
```
The approved RTMR registry must be updated for each new build. This is a one-time operational procedure per image version.

### Q6: Non-Deterministic Inference and Selection Bias — DECIDED: Verifiable Randomness

**The problem**: LLM inference with temperature > 0 is stochastic. A runner who wins the auction gets a 2-hour execution window. On GPU (~4 min/run), they could run inference ~30 times and cherry-pick the result they prefer. On CPU (~22 min/run), they get ~5 attempts. A runner who is also a referral code holder might cherry-pick high commission rates.

**Decision**: Use verifiable on-chain randomness to seed inference.

**Mechanism**:
1. When `closeAuction()` is called, the contract records `block.prevrandao` (beacon chain randomness from that block)
2. The seed is computed as: `seed = keccak256(inputHash || prevrandao)` — included in `_computeInputHash()` or stored separately
3. The enclave uses this seed for llama.cpp's RNG (via `--seed` parameter)
4. The seed is part of the attested input (bound via REPORTDATA), so the contract can verify the enclave used the correct seed
5. The runner cannot predict `prevrandao` at bid time (it depends on the beacon chain's RANDAO reveal in the block that includes `closeAuction()`), and cannot change it after

**Why this works**: `block.prevrandao` is determined by the Ethereum beacon chain validators, not by the runner or the Base sequencer. On L2s like Base, `prevrandao` is inherited from the L1 beacon chain. The runner would need to control the L1 validator set to manipulate it — infeasible.

**Implementation in llama.cpp**: Pass `--seed <uint32>` to the llama-server, or set it per-request in the API call: `{"seed": <int>, ...}`. The enclave computes the seed from the verified input hash + prevrandao and passes it to llama.cpp. With a fixed seed, identical inputs produce identical outputs — the runner gets exactly one possible result.

**Nuance**: The seed makes inference deterministic for a given input, but different seeds produce different outputs. This preserves variety across epochs (each epoch has different prevrandao) while eliminating within-epoch selection bias.

### Q7: Model Loading and Integrity — DECIDED: Load from Disk, Verify Hash

**The question**: Can the binary load the model from a file on disk (provided by the runner) and verify its hash against a hash baked into the binary?

**Yes, this is secure.** The security argument:

1. The binary (enclave image) contains a hardcoded expected hash: `MODEL_SHA256="0b319bd..."`
2. At boot, the binary reads the model file from disk, computes `sha256sum(file)`, and compares against the expected hash
3. If mismatch → abort (no inference, no quote generated)
4. The binary itself is attested via RTMR[0..2] — any change to the expected hash changes the measurements

**The runner's role**: The runner is responsible for making the model file available on disk before booting the TEE. How they obtain it (download from HuggingFace, copy from local storage, torrent, etc.) is irrelevant — the binary only cares that `sha256(file) == expected_hash`. The runner CANNOT substitute a different model because the hash won't match.

**This is strictly better than runtime download** because:
- No network dependency at boot (eliminates DoS via network blocking)
- Faster boot time (no 9 GB download)
- Simpler TCB (no `curl` or HTTP client needed in the image)
- The runner can cache the model across epochs

**Recommended design for production**:
- The enclave image does NOT download anything at runtime
- The model file is mounted into the TEE as a read-only volume (or copied in before boot)
- The entrypoint script verifies `sha256sum(model_path) == EXPECTED_HASH` before starting inference
- The expected hash is a constant in the image (covered by RTMR measurements)
- The runner publishes the model hash in their documentation so other runners know what file to obtain

This makes the enclave truly offline after boot: no network access needed, no HTTP server for model download, minimal attack surface.

## 9. Adversarial Review Passes

### Pass 1: Can a Runner Steal Treasury?

The contract enforces hard bounds: max 10% donated per epoch, bounty capped at 2% of treasury. Even a fully compromised attestation system can only cause the contract to execute bounded actions. The worst case per epoch is a 10% donation to a legitimate nonprofit address (hardcoded, cannot be changed). Over 10 epochs, an attacker could cause ~65% of the treasury to be donated — but only to the three hardcoded nonprofit addresses. The attacker cannot redirect funds to their own address.

**Verdict**: The contract's action bounds are the ultimate safety net. Even total TEE failure doesn't allow theft.

### Pass 2: Can a Runner Grief the System Profitably?

A runner wins the auction at cost `bid + 20% bond`, then intentionally doesn't submit, forfeiting the bond. Cost to attacker: `bond = 0.2 * bid`. Benefit: the epoch is skipped (no donation made, no parameters changed). A well-funded attacker could repeatedly win auctions and grief to prevent the fund from ever donating.

**Defense**: Auto-escalation raises the bid ceiling, making each successive grief more expensive. If the attacker stops, a legitimate runner claims the higher bounty. The economic cost of sustained griefing grows exponentially (10% compounding per epoch).

**Verdict**: Expensive to sustain, self-healing via auto-escalation. Acceptable risk.

### Pass 3: Can a Runner Manipulate the Model via Timing?

A runner could monitor when specific conditions exist (e.g., high treasury balance) and only participate when the model is more likely to produce high-donation actions. This is not an attack on the TEE — it's a market-timing strategy.

**Verdict**: This is a feature, not a bug. Runners self-select epochs where the bounty is worth their cost. The model still makes independent decisions.

### Pass 4: Can Two Colluding Runners Exploit the Auction?

In a sealed-bid first-price auction, colluding runners could agree to bid high (extracting more from the treasury) while avoiding undercutting each other. With only 2-3 runners, cartel behavior is feasible.

**Defense**: The agent can lower `max_bid` in response to consistently high winning bids. New runners can enter permissionlessly, breaking the cartel.

**Verdict**: Cartel risk is real in thin markets. The open auction design (bids visible on-chain) actually helps here — any outsider can see the cartel's bids and undercut them.

### Pass 5: Can a Runner Exploit the Long-Running Enclave?

The enclave runs as a persistent Flask server. The runner sends requests to `/run_epoch`. Between requests, the runner has full control of the host and network. Could a runner:

- **Modify the enclave's memory between requests?** No — TDX isolates the TD's memory from the host. The host VMM cannot read or write TD-private memory.
- **Inject data via the Flask API?** The `/run_epoch` endpoint accepts `epoch_context` (string) and `input_hash` (hex string). The runner fully controls these. This is by design — the REPORTDATA binding ensures the contract can verify what inputs were used. But it means the enclave is NOT "totally offline" in the strictest sense.
- **Exploit Flask/Python vulnerabilities to break out of the application sandbox within the TEE?** The enclave runs as root inside the TD. Flask/Python exploits could allow arbitrary code execution *within* the TEE, which would bypass the application logic while the RTMR measurements remain valid (they were computed at boot, not at request time). However, REPORTDATA is computed by the application code, so the attacker would need to reverse-engineer the hash computation to produce valid REPORTDATA for fabricated outputs — possible but non-trivial within the execution window.

**Stronger design**: Make the enclave a one-shot process that: (1) reads inputs from a file or stdin, (2) runs inference, (3) outputs results, (4) exits. This eliminates the persistent attack surface. The runner would boot a fresh TEE for each epoch.

**Verdict**: The persistent server model has a larger attack surface than a one-shot design. For production, consider switching to a one-shot model where the enclave boots, processes one epoch, and terminates.

### Pass 6: Is the Enclave Truly "Offline"?

The stated goal is that the binary should be "totally offline." Currently it is NOT:
- The model is downloaded from Hugging Face at boot (network dependency)
- The Flask server accepts HTTP requests from the runner (network input)
- The enclave calls the dstack Unix socket for attestation (local IPC)

The model download is verified via SHA-256, so network tampering is caught. The Flask API is the intended input channel (controlled by REPORTDATA binding). The dstack socket is a trusted platform service.

A truly offline design would:
1. Bake the model into the image (no network download)
2. Accept inputs via a mounted file or environment variable (no HTTP server)
3. Write outputs to a file or stdout
4. Boot → compute → attest → exit

This would reduce the TCB (Trusted Computing Base) significantly by eliminating Flask, the HTTP stack, and the runtime download logic. The tradeoff is a larger image (~9 GB with the 14B model baked in) and the need to boot a fresh TEE for each epoch.

**Verdict**: The current design is defensible but not minimal. A one-shot offline design would be strictly stronger from a security perspective.

### Pass 7: Hash Function Mismatch Between Contract and Enclave

The contract computes `inputHash` using `keccak256(abi.encode(...))` (Ethereum's native hash). The enclave computes REPORTDATA using `SHA-256`. These are different hash functions, which is fine — but the interaction needs care:

- `inputHash` (keccak256, 32 bytes) is an opaque blob to the enclave — it just includes it in the SHA-256 chain
- The enclave hashes: `SHA256(inputHash_keccak || SHA256(action) || SHA256(reasoning))` → 32 bytes, zero-padded to 64 bytes
- TDX hardware places these 64 bytes verbatim into the quote's REPORTDATA field (no additional hashing in dstack v0.5.x or bare-metal TDX)

The contract must replicate this exact chain using Solidity's `sha256()` precompile (NOT `keccak256`). The `keccak256` is only used for the initial input hash; all REPORTDATA verification uses SHA-256.

**Verdict**: Must be implemented carefully. An integration test that generates a real quote and verifies it on-chain (or in a Foundry test with a mock) is essential.

### Pass 8: Randomness as an Unattested Input

LLM inference with temperature > 0 uses random number generation. This randomness is an input that is NOT covered by the input hash or REPORTDATA. A runner who controls the RNG seed can influence the model's output.

Inside TDX, the guest has access to `RDRAND`/`RDSEED` CPU instructions which produce hardware random numbers. These are trusted (generated by the CPU, not the host). However, llama.cpp's sampling code may use software PRNGs seeded from various sources — the exact seeding behavior depends on the llama.cpp version.

**Options**:
1. **temperature=0**: Eliminates randomness entirely. Output is deterministic for a given input. Strongest security, but may produce less interesting/varied reasoning.
2. **Attested seed**: Derive the seed inside the enclave from `RDRAND` (hardware RNG, trusted within TDX). The seed becomes part of REPORTDATA. The runner can't control it because they can't influence `RDRAND` from outside the TD.
3. **Verifiable external seed**: Derive the seed from on-chain randomness (e.g., `block.prevrandao`), passed as part of the input. The runner can't predict it at bid time.

**Decision**: Option 3 (verifiable external seed via `block.prevrandao`). See Q6 for full mechanism. With a fixed seed derived from on-chain randomness, inference is deterministic — the runner gets exactly one possible output, eliminating selection bias entirely.
